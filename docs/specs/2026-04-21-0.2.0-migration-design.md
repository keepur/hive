# 0.1.x → 0.2.0 Migration — Design Spec

**Date**: 2026-04-21
**Ticket**: [KPR-55](https://linear.app/keepur/issue/KPR-55) (Phase 5 of [KPR-50](https://linear.app/keepur/issue/KPR-50) — Instance dir restructure, v0.2.0)
**Depends on**: [KPR-51](https://linear.app/keepur/issue/KPR-51), [KPR-52](https://linear.app/keepur/issue/KPR-52), [KPR-53](https://linear.app/keepur/issue/KPR-53), [KPR-54](https://linear.app/keepur/issue/KPR-54)
**Status**: Draft

## Problem

Phases 1-4 define the new layout and the fresh-install story. They do not address the two instances that are already running on May's Mac Mini:

- **dodi** — ~539MB total (447MB `node_modules`, 21MB `.playwright-mcp/`, ~55 loose agent-generated files at the root), currently deployed from the `deploy` branch clone. Mission-critical.
- **keepur** — smaller footprint, same layout. Also on the `deploy` branch clone path.

Plus any beta customers already running `v0.1.x` whose instances need to cut over cleanly to `0.2.0`.

Phase 5 is the one-shot migration that takes an existing 0.1.x instance and reshapes it into the 0.2.0 layout *without* data loss and with a clean rollback path.

## Goal

A single command — `hive migrate-0.2` — that, run against a running or stopped 0.1.x instance dir, produces a clean 0.2.0 instance:

```
~/services/hive/dodi/                     (before)          (after)
├── dist/                                 engine files     → .hive/pkg/ etc.
├── node_modules/                         installed deps   → wiped (new .hive/ has pkg bundle)
├── src/                                  source           → wiped (shipped inside .hive/ only if present)
├── plugins/claude-code/                  engine plugins   → .hive/plugins/claude-code/
├── scripts/, setup/, seeds/, ...         engine assets    → .hive/ or inside pkg bundle
├── .git/                                 git state        → deleted
├── .hive/git/                            instance-git     → .hive-state/git/ (Phase 2 rename)
├── 55+ loose agent files at root         agent outputs    → agents/<id>/reports/archive-pre-0.2/
├── .playwright-mcp/                      browser logs    → deleted (21MB of cruft)
├── hive.yaml, .env, logs/, beekeeper.yaml                 → unchanged (survive)
├── .hive-generated.json                                   → unchanged (survive)
├── (no agents/, workflow/, data/, skills/, plugins/ yet)  → created empty (from Phase 1)
└── (no backup)                                            → ../<instance>.pre-0.2-bak (rollback snapshot)
```

Ordering and risk budget:

- **keepur canary first.** Smaller, less observed blast radius. Stay on it for 48 hours before migrating dodi.
- **dodi 48h later.** Only if keepur is clean.
- **Beta customers last.** Only after dodi is stable for another week; ship `hive migrate-0.2` as a documented one-liner in the 0.2.0 release notes.

**Shared HIVE_HOME note (dodi + personal).** The `personal` instance on May's Mac Mini shares `HIVE_HOME=~/services/hive/dodi` with dodi (per `instances.conf`: `dodi|hive.yaml|...` and `personal|hive-personal.yaml|...` both resolve against the same dir). Migrating dodi simultaneously migrates the filesystem layout for personal. This is correct — both instances' engine lives in the shared `.hive/` after migration, both instances' live plists at shared `<dir>/service/` — but it means running `hive migrate-0.2 ~/services/hive/dodi` takes *both* instances offline for the duration. Communicate accordingly.

Non-goals:
- Changing any MongoDB data. Agent definitions, memory, sessions, callbacks stay where they are.
- Migrating or renaming any agent. The migration is filesystem-only.
- Forking a separate 0.2.0 branch — by the time migration runs, 0.2.0 is published on npm.

## Design

### Migration command

`hive migrate-0.2` is a new CLI subcommand in `src/cli/migrate.ts`. It must run from the *old* layout (there's no 0.2.0 engine inside `.hive/` yet) so it's delivered as a **standalone standalone script** downloaded from `hive-docs`, not bundled in the 0.2.0 engine that the migration would install.

```
curl -fsSL https://raw.githubusercontent.com/keepur/hive-docs/main/install/migrate-0.2.sh | bash -s -- ~/services/hive/dodi
```

The script:

1. Validates the target dir is a 0.1.x layout (checks for `dist/index.js`, `node_modules/`, a `hive.yaml` at root, absence of a populated `.hive/pkg/`).
2. Stops the launchd service for the instance.
3. Creates a snapshot: `cp -a <instance> <instance>.pre-0.2-bak`. Fails loud if the bak already exists (no accidental double-migrate).
4. Runs the migration steps below.
5. Runs `npm i -g @keepur/hive@0.2.0` if the global CLI isn't already on 0.2.0.
6. Invokes the new 0.2.0 CLI's `hive doctor` to sanity-check the result.
7. Restarts the launchd service.
8. Posts a summary to Slack via the same notification path `deploy.sh` uses.

Implementation language: bash only (plus `yq` for YAML surgery in Step 8, `rsync` for Step 7 file copies). No TypeScript, no node runtime dependency — the script has to run before the 0.2.0 engine is installed, and shipping migration logic *inside* the engine it's migrating from would require running the new engine against the old layout, which is exactly the bootstrapping problem we want to avoid. No `jq` — nothing parses JSON.

**Idempotence.** Re-running against an already-migrated dir detects the state at step 1 and exits 0 with "already on 0.2.0."

### Migration steps (what the script does)

Each step is a separate bash function. All paths are under `$INSTANCE_DIR`.

#### Step 1 — Pre-flight

- Set `INSTANCE_ID="$(basename "$INSTANCE_DIR")"` — used later in Step 8 yq rewrite. This is the canonical per-instance directory name under `~/services/hive/`; it's also what `instances.conf` uses as the `ID` column. Derivation is intentional: the script takes only one positional arg (`$INSTANCE_DIR`), and `INSTANCE_ID` must be unambiguous for path rewrites.
- Assert `hive.yaml` exists.
- Assert `dist/index.js` *or* `.hive/git/` exists (evidence of 0.1.x layout). If neither: the instance is already migrated or in an unknown state; error out.
- Assert `$INSTANCE_DIR.pre-0.2-bak` does not exist. If it does: "existing snapshot found — rename or delete before retrying."
- Assert the launchd service for this instance is either not loaded or stopped. If it's running, offer to stop it (`launchctl bootout`), require confirmation.
- Assert disk space: we need ~2× the instance dir size during snapshot. `df` check, exit with clear error if < 2× free.
- Assert required CLIs are installed: `yq` (used by Step 8), `rsync` (used by Step 7), `realpath` and `readlink` (used by Step 5 and Step 12 for plist discovery). All four ship on a default macOS install except `yq` (Homebrew). If `yq` missing: print `"yq required — install with: brew install yq"` and exit 1. The preflight must catch this *before* Step 2's snapshot is taken, to avoid leaving an orphan backup.

#### Step 2 — Snapshot for rollback

```bash
cp -a "$INSTANCE_DIR" "$INSTANCE_DIR.pre-0.2-bak"
```

This is the rollback anchor. It copies the entire instance dir including the in-flight `node_modules/` and git state. ~500MB for dodi, few seconds on SSD.

#### Step 3 — Create the new skeleton namespaces

```bash
mkdir -p "$INSTANCE_DIR/agents" "$INSTANCE_DIR/workflow" "$INSTANCE_DIR/data" "$INSTANCE_DIR/skills" "$INSTANCE_DIR/plugins"
mkdir -p "$INSTANCE_DIR/.hive" "$INSTANCE_DIR/.hive-state"
```

Empty dirs are fine — Phase 1's per-agent cwd resolver creates `agents/<id>/scratch/` on first use.

#### Step 4 — Relocate `.hive/git/` → `.hive-state/git/`

The destination names (`.hive-state/`, `installed-snapshot.json`, `previous-snapshot.json`, `upgrade-notice-emitted`) must match Phase 2's `hiveStateDir` and `instanceGitDir` exports in `src/paths.ts` exactly — this migration writes to the paths the new engine will read from on first boot. Verify against the Phase 2 spec (`docs/specs/2026-04-21-engine-hive-dir-design.md`, section "Code changes → `src/paths.ts`") before running.

```bash
if [[ -d "$INSTANCE_DIR/.hive/git" ]]; then
  mv "$INSTANCE_DIR/.hive/git" "$INSTANCE_DIR/.hive-state/git"
fi
# Also move the snapshots if present at the old .hive/ path
for f in installed-snapshot.json previous-snapshot.json upgrade-notice-emitted; do
  [[ -e "$INSTANCE_DIR/.hive/$f" ]] && mv "$INSTANCE_DIR/.hive/$f" "$INSTANCE_DIR/.hive-state/$f"
done
```

After this, `.hive/` is empty or absent. The engine population in step 7 drops into a clean slot.

#### Step 5 — Remove engine code from the instance root

These files and dirs were engine content under the 0.1.x model; they're about to be replaced by the tarball extract in step 7. Delete them now to avoid leftover cruft:

```bash
rm -rf \
  "$INSTANCE_DIR/dist" \
  "$INSTANCE_DIR/node_modules" \
  "$INSTANCE_DIR/src" \
  "$INSTANCE_DIR/build" \
  "$INSTANCE_DIR/setup" \
  "$INSTANCE_DIR/seeds" \
  "$INSTANCE_DIR/templates" \
  "$INSTANCE_DIR/tests" \
  "$INSTANCE_DIR/scripts" \
  "$INSTANCE_DIR/docs" \
  "$INSTANCE_DIR/install" \
  "$INSTANCE_DIR/plugins/claude-code" \
  "$INSTANCE_DIR/package.json" \
  "$INSTANCE_DIR/package-lock.json" \
  "$INSTANCE_DIR/tsconfig.json" \
  "$INSTANCE_DIR/tsconfig.plugins.json" \
  "$INSTANCE_DIR/eslint.config.js" \
  "$INSTANCE_DIR/vitest.config.ts" \
  "$INSTANCE_DIR/AGENTS.md" \
  "$INSTANCE_DIR/README.md" \
  "$INSTANCE_DIR/CLAUDE.md" \
  "$INSTANCE_DIR/.github"

# Inside service/: delete every engine-shipped script and template plist, but
# preserve any plist that is currently the target of a ~/Library/LaunchAgents/
# symlink — that's the live plist for one of the instances sharing this HIVE_HOME.
#
# Hardcoding filenames is brittle: dodi's LaunchAgent label is `com.hive.agent`
# (legacy, not `com.hive.dodi.agent`), and the `personal` instance shares this
# HIVE_HOME with dodi (its label is `com.hive.personal.agent`, also living in
# the same service/ dir). A blanket whitelist misses this. Detect at runtime:

declare -a LIVE_PLISTS=()
for link in "$HOME/Library/LaunchAgents"/com.hive.*.plist; do
  [[ -L "$link" ]] || continue
  target=$(readlink "$link" 2>/dev/null || true)
  # readlink returns absolute or relative; realpath normalizes
  abs=$(realpath "$link" 2>/dev/null || true)
  if [[ -n "$abs" && "$abs" == "$INSTANCE_DIR/service/"* ]]; then
    LIVE_PLISTS+=("$(basename "$abs")")
  fi
done

for f in "$INSTANCE_DIR/service"/*; do
  [[ -e "$f" ]] || continue
  name=$(basename "$f")
  is_live=false
  for live in "${LIVE_PLISTS[@]:-}"; do
    [[ "$name" == "$live" ]] && is_live=true && break
  done
  if ! $is_live; then
    rm -f "$f"
  fi
done
# The `service/` dir itself, and any live plist inside it, survives.
# Step 10 regenerates the live plist in place to pick up the new
# ProgramArguments (.hive/pkg/server.min.js).
```

The `rm`s are enumerated explicitly — no wildcard globbing — so a typo in `hive.yaml` or `.env` can't get them caught. `service/` itself is preserved (the live plist lives there and is the LaunchAgents symlink target); only the engine-shipped scripts and template plists inside `service/` are removed.

#### Step 6 — Classify and relocate loose agent files

The `~55 loose agent files` at dodi's root need to move somewhere — either archived per-agent (preferred) or dumped into an unsorted bucket (fallback).

Classification by filename prefix, checked in order. First match wins:

| Pattern | Destination | Rationale |
|---------|-------------|-----------|
| `milo-*`, `MILO-*` | `agents/milo/reports/archive-pre-0.2/` | Obvious agent ownership |
| `river-*`, `RIVER-*` | `agents/river/reports/archive-pre-0.2/` | |
| `jessica-*`, `JESSICA-*` | `agents/jessica/reports/archive-pre-0.2/` | |
| `wyatt-*`, `WYATT-*` | `agents/wyatt/reports/archive-pre-0.2/` | |
| `rae-*`, `RAE-*` | `agents/rae/reports/archive-pre-0.2/` | |
| `chloe-*`, `CHLOE-*` | `agents/chloe/reports/archive-pre-0.2/` | |
| `colt-*`, `COLT-*` | `agents/colt/reports/archive-pre-0.2/` | |
| `sige-*`, `SIGE-*` | `agents/sige/reports/archive-pre-0.2/` | |
| `fb-*.md`, `linkedin-*.md`, `x-*.md`, `x-snapshot-*` | `data/archive-pre-0.2/social-scrapes/` | Browser scrapes — no single agent owner |
| `*-standup-*.{md,json}`, `sales-*-standup-*` | `agents/milo/reports/archive-pre-0.2/` | Standups are Milo's domain; verified against dodi state |
| `PERMIT-*`, `river-*-permits-*.csv`, `high-tier-permits*.csv`, `*-permits.csv` | `agents/river/reports/archive-pre-0.2/` | Permit data is River's |
| `HUBSPOT-*`, `STALE-DEALS-*`, `stale-deals.csv`, `*-sales-pipeline-*` | `agents/milo/reports/archive-pre-0.2/` | Pipeline data → Milo |
| `LEAD-SEGMENTATION-*`, `QUERY-RESULTS-*` | `data/archive-pre-0.2/unsorted/` | Could be Milo or River; safer to park |
| `*.ts` (ad-hoc scripts at root — analyze-*, check-*, create-*, extract-*, fetch-*, get-*, verify-*) | `data/archive-pre-0.2/scripts/` | Not deleted — some may be referenced in tickets. Archive for possible reference. |
| `README-*.md` | `data/archive-pre-0.2/` | Per-artifact readmes; keep alongside data |
| `.playwright-mcp/` | deleted outright | 21MB of console logs; rolling archive, no long-term value. Dry-run reports size before delete. |
| (anything else unmatched) | `data/archive-pre-0.2/unsorted/` | Catch-all |

The classifier is a simple bash `case`/glob loop. **Ordering is load-bearing**: agent-prefix patterns (`milo-*`, `river-*`, etc.) must be checked *before* content-type patterns (`*-permits.csv`, `stale-deals.csv`). Example: `river-high-tier-permits-extraction.csv` matches both `river-*` (→ River's reports) and `*-permits.csv` (→ River's reports by a different path). The agent-prefix rule fires first and wins; the destinations happen to agree here, but for files like `HUBSPOT-river-pulls.csv` the same ordering rule becomes decisive (`HUBSPOT-*` → Milo wins over any later River-specific pattern). Implement as `case` arms top-to-bottom matching the table's row order.

**Dry-run mode**: `hive migrate-0.2 --dry-run <instance>` runs *only* step 1 (preflight validation) and emits the classifier table for every loose file. All filesystem mutations — `cp -a` snapshot, `mkdir` skeleton, `mv` renames, `rm` deletions, yaml surgery, engine populate, daemon regenerate — are suppressed. The live instance is not touched in any way. This is the one invariant worth flagging to implementers: a partial dry-run that creates `agents/<id>/` stubs "just to test" would break idempotence of a subsequent real run. Keep it read-only end-to-end.

#### Step 7 — Populate `.hive/` with the 0.2.0 engine

```bash
# Preferred — uses the global npm install.
HIVE_VERSION=0.2.0 npm i -g @keepur/hive
CLI_ROOT=$(dirname "$(realpath "$(which hive)")")/..

# Mirrors Phase 4's populateEngine from shell. The entry list is the npm-pack
# tarball shape that Phase 3's deploy flow populates into .hive/ for upgrades;
# it is NOT the Phase 2 "repo move" shape (dist/, node_modules/, src/).
# Phase 3 changed the deployed .hive/ contents from repo-shape to pack-shape,
# and Phase 5's migration targets the post-Phase-3 state directly.
# Exact list must match Phase 4's PACKAGE_ENTRIES in src/setup/populate-engine.ts.
for entry in pkg seeds templates scripts/honeypot package.json; do
  src="$CLI_ROOT/$entry"
  if [[ ! -e "$src" ]]; then
    # Not a silent skip: every entry in this list must exist in a valid
    # 0.2.0 install. If one is missing, the global CLI is broken or the
    # list has drifted from PACKAGE_ENTRIES. Fail loud.
    echo "ERROR: expected engine entry $entry missing from $CLI_ROOT. " \
         "Verify @keepur/hive@0.2.0 installed correctly: npm ls -g @keepur/hive" >&2
    exit 1
  fi
  dst="$INSTANCE_DIR/.hive/$entry"
  mkdir -p "$(dirname "$dst")"
  if [[ -d "$src" ]]; then
    rsync -a "$src/" "$dst/"
  else
    rsync -a "$src" "$dst"
  fi
done
```

The alternative — `npm pack @keepur/hive@0.2.0` + tarball extract — also works and produces the same `.hive/` contents. The shell loop above is preferred because the global install is already required for Step 10 (`hive daemon start`).

#### Step 8 — Rewrite `hive.yaml` paths

`codeTask.pluginDirs` at dodi today points at `~/services/hive/plugins/claude-code/...` (a shared dir across instances). After migration each instance has its own engine plugins at `<instance>/.hive/plugins/claude-code/`. The script rewrites the yaml in place:

```bash
# Using yq (prerequisite; the migrate script checks for it before running).
# The array constructor [...] wrapping the iteration is LOAD-BEARING:
# without it, yq assigns the stream result to the target, which collapses
# to the last scalar only — overwriting the array with a single string.
yq -i '.codeTask.pluginDirs = [
  .codeTask.pluginDirs[]
  | sub("^~/services/hive/plugins/"; "~/services/hive/'"$INSTANCE_ID"'/.hive/plugins/")
]' "$INSTANCE_DIR/hive.yaml"
```

Notes:

- `~` is **not** expanded by yq — the regex operates against the literal yaml value, which stores the tilde. This is correct: the yaml stores unexpanded paths so they resolve at config-load time against the actual `$HOME`. Don't `realpath` before matching.
- The sub pattern is anchored at `^` so only the leading segment matches; a reference to a nested dir like `plugins/claude-code/dodi-dev` becomes `.hive/plugins/claude-code/dodi-dev` correctly.
- If any `pluginDirs` entry doesn't start with `~/services/hive/plugins/` (e.g., an operator pointed at an absolute path elsewhere), it passes through unchanged — also correct.

This is the one yaml surgery the script performs. Anything else in `hive.yaml` stays byte-identical — business config, slack config, etc. are layout-independent.

If `yq` is missing, the script halts with a clear "install yq via `brew install yq` and retry" message. (Preflight check catches this at step 1.)

#### Step 9 — Remove instance-root `.git/`

```bash
rm -rf "$INSTANCE_DIR/.git"
```

Instance is no longer a git clone. Phase 3 already stopped writing to this dir; Phase 5 completes the removal.

#### Step 10 — Regenerate live plist(s)

A single HIVE_HOME may host multiple labels — dodi + personal share `~/services/hive/dodi/` (and thus share `service/`). Each label has its own plist, and each needs the new `ProgramArguments` pointing at `.hive/pkg/server.min.js` (or `.hive/dist/index.js` fallback per Phase 2). Two plist kinds to handle:

1. **Primary / generated plist** (e.g., dodi's `com.hive.agent.plist`): regenerated by the CLI.

   ```bash
   HIVE_HOME="$INSTANCE_DIR" hive daemon start
   ```

   This writes `<instance>/service/<primary-label>.plist` with the new ProgramArguments and (re-)creates the `~/Library/LaunchAgents/` symlink.

2. **Secondary / hand-authored plists** (e.g., personal's `com.hive.personal.agent.plist`): not managed by `hive daemon start`. The Phase 2 engine ships an updated template at `.hive/service/com.hive.personal.agent.plist` with the `.hive/dist/index.js` ProgramArguments baked in. Copy it into the live location, then bootstrap:

   ```bash
   # Loop over every live plist in service/ that is NOT the primary label
   # (primary is the one hive daemon start just wrote). For each, overwrite
   # with the engine's shipped template and re-bootstrap.
   for live_plist in "${LIVE_PLISTS[@]:-}"; do
     # LIVE_PLISTS was populated in Step 5 preflight. Skip the primary we
     # just regenerated via hive daemon start.
     [[ "$live_plist" == "<primary>.plist" ]] && continue
     src="$INSTANCE_DIR/.hive/service/$live_plist"
     dst="$INSTANCE_DIR/service/$live_plist"
     if [[ -f "$src" ]]; then
       cp "$src" "$dst"
       label="$(basename "$live_plist" .plist)"
       launchctl bootout "gui/$(id -u)/$label" 2>/dev/null || true
       launchctl bootstrap "gui/$(id -u)" "$HOME/Library/LaunchAgents/$live_plist"
     else
       echo "WARNING: no engine-shipped template for $live_plist at $src; " \
            "this plist was not updated. Verify ProgramArguments by hand." >&2
     fi
   done
   ```

   Resolving "which is the primary label?" uses the same logic as `hive daemon start` — the label derived from the instance's `hive.yaml` `slack.label` / CLI default. A simpler heuristic: the primary is the plist `hive daemon start` wrote in pass 1; enumerate `service/*.plist` mtimes after pass 1 and treat the newest as primary, the rest as secondaries.

Shared-HIVE_HOME case (dodi + personal): Step 10 runs `hive daemon start` once (regenerates dodi's primary plist), then the secondary loop picks up `com.hive.personal.agent.plist` and overwrites it from the engine template. Both plists point at the new `.hive/` engine; both labels are bootstrapped; both instances come back online.

#### Step 11 — Health check

Same `health_check` function as `deploy.sh` — tail the log for `"Hive is running"` within 30s. On failure, auto-rollback (step 12). On success, post the success message to Slack.

#### Step 12 — Auto-rollback on failure

```bash
if ! health_check; then
  # Discover the actual launchd label(s) attached to this instance dir.
  # Don't assume com.hive.$INSTANCE_ID.agent — dodi's legacy label is
  # com.hive.agent, and a shared HIVE_HOME may host multiple labels
  # (dodi + personal). Bootout every label whose plist is a symlink into
  # $INSTANCE_DIR/service/.
  declare -a LABELS=()
  for link in "$HOME/Library/LaunchAgents"/com.hive.*.plist; do
    [[ -L "$link" ]] || continue
    abs=$(realpath "$link" 2>/dev/null || true)
    if [[ -n "$abs" && "$abs" == "$INSTANCE_DIR/service/"* ]]; then
      LABELS+=("$(basename "$link" .plist)")
    fi
  done

  for label in "${LABELS[@]:-}"; do
    launchctl bootout "gui/$(id -u)/$label" 2>/dev/null || true
  done

  rm -rf "$INSTANCE_DIR"
  mv "$INSTANCE_DIR.pre-0.2-bak" "$INSTANCE_DIR"

  # Re-bootstrap every label we booted out. The symlinks at
  # ~/Library/LaunchAgents/ now point back at the restored snapshot's plists.
  for label in "${LABELS[@]:-}"; do
    launchctl bootstrap "gui/$(id -u)" "$HOME/Library/LaunchAgents/$label.plist"
  done

  notify "Migration to 0.2.0 FAILED and was rolled back for $INSTANCE_DIR. Instance(s) back on 0.1.x: ${LABELS[*]:-<none>}."
  exit 1
fi
```

The snapshot at `<instance>.pre-0.2-bak` is the rollback anchor. If health check fails, the *failed* `<instance>/` is deleted and the backup is renamed into its place. The LaunchAgent plist at `~/Library/LaunchAgents/` points at a path *inside* the instance dir (e.g., `<instance>/service/com.hive.agent.plist`); after `mv bak <instance>`, the symlink target exists again and re-bootstrapping from it restores the old configuration exactly.

**Label discovery** uses the same symlink-readlink trick as Step 5 — never hardcode `com.hive.$INSTANCE_ID.agent`, since:
- dodi's label is `com.hive.agent` (legacy), not `com.hive.dodi.agent`.
- A shared HIVE_HOME (dodi + personal) has multiple labels rooted in the same dir; both must be booted out and re-bootstrapped atomically.

### Rollout sequencing

**keepur canary (day 0)**

1. `hive migrate-0.2 --dry-run ~/services/hive/keepur` — review classifier output, add patterns if needed.
2. `hive migrate-0.2 ~/services/hive/keepur` — real run.
3. Monitor for 48h: Slack errors, agent responsiveness, log volume, any `hive doctor` regressions.
4. **Gate**: keepur is green at 48h → proceed to dodi. Not green → fix forward and re-evaluate before dodi.

**dodi migration (day 2)**

1. Dry-run, apply same patterns as keepur.
2. Live run during a low-activity window (early morning, before May's workday).
3. Watch for 24h. Escalation path: `hive rollback` (Phase 3) for engine-level issues; full bak-snapshot restore for migration-level issues.

**Beta customer self-migration (day 9+)**

1. Once dodi is stable, publish the migration one-liner in the 0.2.0 release notes.
2. `hive-docs` gets a `docs/migrating-to-0.2.md` walkthrough.
3. Customers run it on their own schedule. `beta@keepur.io` handles escalations.

### Downtime estimate

Per-instance:

- Service stop: < 5s
- Snapshot (cp -a): ~5s on SSD for 500MB
- Rm engine files: < 1s
- Classifier loop: < 1s for ~60 files
- npm install 0.2.0 (if CLI not already on it): 15-30s depending on network
- Engine populate: ~2s
- Daemon start + health check: up to 35s

Total: ~1-2 minutes per instance. Budget 5 minutes per instance for comms headroom.

Communicated in the release notes: "You'll be offline for about 5 minutes during migration."

## Runtime failure modes

1. **`cp -a` fails midway (disk full, permission error).** Caught by `set -e`. Partial bak left behind. Operator error path: `rm -rf <instance>.pre-0.2-bak` and retry.
2. **Snapshot succeeds but engine wipe leaves dangling symlinks or junk.** The `rm`s are enumerated explicitly — anything not listed stays, which is conservative. If a new file appeared in 0.1.x that's not in the `rm` list, it stays at the instance root and is flagged in the post-migration `hive doctor` check as "unexpected file."
3. **`npm i -g @keepur/hive@0.2.0` fails (network, registry outage).** Migration halts before step 7, after snapshot. Operator can roll back via `mv <bak> <instance>` and retry when network returns.
4. **`yq` not installed.** Preflight (step 1) catches this with a clear install prompt. No partial state.
5. **Classifier miscategorizes a file.** Worst case: file lands in `data/archive-pre-0.2/unsorted/`. Files are never deleted by the classifier; only `.playwright-mcp/` is deleted, and that's called out explicitly.
6. **Agent has a thread in flight when migration starts.** Service stop at step 1 interrupts it. The SDK's in-flight query aborts gracefully (agent-runner's existing timeout path). On restart, any pending Slack message hits the 60s dedup window and is picked up again. No message loss expected; occasional duplicate assistant reply is the risk.

   **Playwright MCP orphan risk.** The Playwright MCP stdio subprocess connects to an external Chrome process via `--cdp-endpoint`. When the parent Hive process SIGTERMs, the Playwright MCP child should terminate along with the rest of the session tree. However, in rare cases (hung child, async file-write in flight), the Playwright MCP process may persist with an open write handle on `.playwright-mcp/` at the moment of `rm -rf .playwright-mcp`. Worst case: the `rm` completes partially and leaves an orphan process writing to phantom paths for a few more seconds. Mitigation: the preflight check at step 1 runs `pgrep -f playwright-mcp` after the launchctl bootout and waits up to 5 seconds for exit; if any remain, the script errors with a manual-kill hint. Low-severity overall — 21MB of expendable console logs is the asset at risk.
7. **`~/Library/LaunchAgents/` symlink target has drifted from the bak snapshot's plist.** The bak contains `<instance>.pre-0.2-bak/service/<label>.plist`. The symlink in `~/Library/LaunchAgents/` still points at `<instance>/service/<label>.plist` (the *path*, not the inode). After rollback (`mv bak <instance>`), the symlink target exists again. Verified works.
8. **User runs migrate on a 0.2.0 instance.** Step 1 preflight detects `.hive/pkg/` already populated and exits "already migrated."
9. **User runs migrate with `HIVE_HOME` set to a different path than the script arg.** The script uses its positional arg explicitly, not `HIVE_HOME`. No confusion.

## Acceptance

- **Dry-run classifier**: on a copy of dodi's current state, `hive migrate-0.2 --dry-run` emits a classification table. Every loose file has a destination. Zero files land in `unsorted/` unexpectedly (the classifier is updated until this holds).
- **keepur canary**: 48h post-migration, `hive doctor` all-green, no Slack errors, no agent responsiveness regressions. Post-migration `ls ~/services/hive/keepur/` shows only: `.hive/`, `.hive-state/`, `.hive-generated.json`, `.env`, `hive.yaml`, `beekeeper.yaml`, `logs/`, `service/`, `agents/`, `workflow/`, `data/`, `skills/`, `plugins/`.
- **dodi migration**: same acceptance as keepur, verified 24h after cutover. `STALE-DEALS-DETAILED.txt` and its 54 loose siblings are under `agents/<id>/reports/archive-pre-0.2/` or `data/archive-pre-0.2/`, not at root.
- **Rollback tested**: synthetically fail a step (e.g., point `HIVE_VERSION` at a non-existent tag); assert auto-rollback fires, assert the `.pre-0.2-bak` restore produces a byte-identical instance to pre-migration, assert launchd-loaded and Slack-reachable.
- **Idempotence**: re-running the migrate script against an already-migrated instance exits 0 with "already on 0.2.0"; no files touched.
- **Beta customer path**: a documented one-liner exists in `hive-docs`; end-to-end test of "install 0.1.10 fresh, make some agent activity, migrate to 0.2.0, verify" passes in an isolated env.

## Open questions

None.

- Migration is a standalone shell script in `hive-docs/install/`, not bundled in the 0.2.0 engine, because it has to run before 0.2.0 is installed.
- Loose-file classification archives per-agent (into `agents/<id>/reports/archive-pre-0.2/`) with a `data/archive-pre-0.2/unsorted/` fallback. Nothing is deleted except `.playwright-mcp/`, which is explicitly noted.
- Agent inference is prefix-based; ambiguous files (`PERMIT-*` could be Milo or River) resolve by the canonical table above, corrected against keepur canary before dodi.
- Downtime budget: ~5 minutes per instance, documented in release notes.
- dodi and keepur use the same classifier and the same script. No per-customer forks.
